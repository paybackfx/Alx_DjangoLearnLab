# Plan d'implémentation : Social Media API

Ce document détaille toutes les étapes requises pour la création de l'API du réseau social, divisé par "tasks" (tâches) comme spécifié.

## 0. Initialisation du Projet & Authentification
Il s'agit de mettre en place la base du projet Django et de gérer les utilisateurs avec le Django REST Framework (DRF).

- [ ] **Configuration du projet**
  - Créer l'environnement virtuel (si nécessaire).
  - Installer Django, le Django REST Framework ET Pillow pour gérer les images : `pip install django djangorestframework Pillow`.
  - Démarrer le projet Django : `django-admin startproject social_media_api`.
  - Créer l'application logicielle pour les comptes : `python manage.py startapp accounts`.
  - Ajouter `rest_framework` et `accounts` aux `INSTALLED_APPS` dans `settings.py`.
- [ ] **Modèle Utilisateur personnalisé**
  - Créer un modèle qui étend `AbstractUser` dans `accounts/models.py`.
  - Ajouter les champs supplémentaires : `bio` (TextField), `profile_picture` (ImageField ou URLField), et `followers` (ManyToManyField référant à 'self' avec `symmetrical=False`).
  - Configurer `AUTH_USER_MODEL` dans `settings.py`.
- [ ] **Authentification par Token**
  - Ajouter `rest_framework.authtoken` aux `INSTALLED_APPS`.
  - Faire les migrations (`makemigrations` et `migrate`) pour le modèle custom et les tokens.
  - Créer des serializers dans `accounts/serializers.py` pour l'enregistrement (Register) et la connexion (Login), s'assurant qu'ils retournent un token.
  - Créer des vues API pour l'enregistrement, le login, et le profil (récupération de profil en lecture/écriture).
- [ ] **Routes & Tests**
  - Configurer `accounts/urls.py` avec les routes `/register`, `/login` et `/profile`.
  - Effectuer les tests de base avec Postman.

## 1. Fonctionnalités de Posts et Commentaires
Mise en place des modèles de contenu pour le réseau social (Création, Lecture, Mise à jour, Suppression).

- [ ] **Création des modèles**
  - Créer une application `posts` (`python manage.py startapp posts`).
  - **Modèle Post** : `author` (ForeignKey vers User), `title` (CharField), `content` (TextField), `created_at` (DateTimeField auto_now_add), `updated_at` (DateTimeField auto_now).
  - **Modèle Comment** : `post` (ForeignKey vers Post), `author` (ForeignKey vers User), `content`, `created_at`, `updated_at`.
  - Appliquer les migrations pour `posts`.
- [ ] **Serializers & ViewSets**
  - Créer `PostSerializer` et `CommentSerializer` qui gèrent les relations de clés étrangères.
  - Créer des ModelViewSets dans `posts/views.py` pour un CRUD complet des Posts et Comments.
  - Implémenter les permissions personnalisées (`IsAuthorOrReadOnly`) pour qu'un utilisateur ne puisse modifier ou supprimer que ses propres contenus.
- [ ] **Pagination et Filtres**
  - Ajouter une classe de pagination pour les lister (ex: `PageNumberPagination`).
  - Permettre le filtrage (SearchFilter/DjangoFilterBackend) sur le `title` ou `content` des posts.
- [ ] **Configuration URLs**
  - Utiliser un `DefaultRouter` du DRF dans `posts/urls.py` pour déclarer les endpoints (ex: `/posts/` et `/comments/`).

## 2. Abonnements (Follows) et Fil d'actualité (Feed)
Mise en œuvre du côté "social" : s'abonner / se désabonner aux autres et voir leur contenu.

- [ ] **Gestion des Abonnements (Follows)**
  - *(Généralement couvert dans le champ `followers` configuré à l'étape 0)*.
  - Créer des points d'accès (views) dans `accounts/views.py` pour suivre ou se désabonner d'un utilisateur (actions `follow_user` et `unfollow_user`).
  - S'assurer que les permissions sont valides pour ces actions.
  - Ajouter le routing tel que `/follow/<int:user_id>/` et `/unfollow/<int:user_id>/`.
- [ ] **Le Feed (Flux)**
  - Dans l'application `posts`, créer une nouvelle vue ou une action dans le ViewSet de Post pour `/feed/`.
  - La requête doit récupérer uniquement les posts créés par les utilisateurs contenus dans `request.user.following.all()`.
  - Ordonner les posts par `created_at` descendant (les plus récents en premier).
- [ ] **Tests & Doc**
  - Tester les relations de suivi. Vérifier sur Postman que le feed retourne correctement les données.

## 3. Likes et Notifications
Gestion des interactions utilisateur (j'aime) et du système pour avertir les usagers de l'activité.

- [ ] **Application des Likes**
  - Dans `posts/models.py`, ajouter le modèle `Like` avec les champs `post` (ForeignKey Post) et `user` (ForeignKey User). Ajout d'une contrainte d'unicité (unique_together) pour empêcher un utilisateur de liker plusieurs fois le même post.
  - Raccorder avec des routes : `/posts/<int:pk>/like/` et `/posts/<int:pk>/unlike/`.
- [ ] **Système de Notifications**
  - Créer une nouvelle app : `notifications`.
  - **Modèle Notification** : `recipient` (User qui la reçoit), `actor` (User qui fait l'action), `verb` (CharField, ex: "a liké votre post"), `target` (GenericForeignKey ou relation simple selon la complexité, référant au Post ou Comment), `timestamp` (auto_now_add), et `is_read` (Boolean).
  - Implémenter la création automatique (ex: depuis les vues de Like, de Follow ou via les Signals de Django).
  - Créer l'endpoint `/notifications/` (Liste, Marquage comme lu).

## 4. Déploiement en Production
Mettre l'API en ligne.

- [ ] **Ajustement des Settings**
  - Configurer `DEBUG = False`.
  - Paramétrer le tuple `ALLOWED_HOSTS`.
  - Variables d'environnement pour stocker les clés secrètes et les configurations de base de données (ex: python-dotenv ou django-environ).
  - Mesures de sécurité supplémentaires (HTTPS, XSS, etc.).
- [ ] **Base de Données & Fichiers statiques**
  - Basculer depuis SQLite vers PostgreSQL.
  - Configurer Gunicorn.
  - Gérer les fichiers statiques (WhiteNoise ou Amazon S3 pour des fichiers media comme les profile pictures).
- [ ] **Hébergement**
  - Choisir l'hébergeur (Heroku, Render, DigitalOcean, etc.).
  - Processus de déploiement (Push GitHub, variables d'environnements distantes).
  - Validation finale sur serveur distant.
